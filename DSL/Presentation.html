Core DSL Objects

DSL Model (ElmDsl)

The instance object that implements your dynamic model in the network

  • Connected to network elements
  • Contains parameter values
  • Executes during simulation

DSL Model Type (BlkDef)

The core definition object that defines model structure and logic

  • Contains the model blueprint
  • Defines equations and logic
  • Reusable template

Think of it as: DSL Model Type is the "class definition", DSL Model is the "instance"

Two Definition Approaches

1. Graphical Method

  • Visual block diagrams
  • Drag-and-drop interface
  • Intuitive for engineers
  • Uses predefined DSL macros
  • Real-time visual feedback

2. Code-Based Method

  • Direct DSL code writing
  • More flexible and powerful
  • Suitable for complex models
  • Text-based equations
  • Advanced customization

Recommendation: Start with graphical method for learning, advance to code-based for complex applications

Main Model Identification

üéØ How to Identify a Main Model

Main Model Path

DSL Model Type (BlkDef) ‚Üí DSL Model (ElmDsl) ‚Üí Composite Model (ElmComp) ‚Üí Network

Key Identification Criteria:

  • Network Instantiation: Instantiated as DSL Model (ElmDsl) in network data
  • Top-Level Position: Highest level in model hierarchy
  • Complete Functionality: Contains full model logic, not partial
  • Initialization Responsibility: Must contain initialization conditions for ALL:
    • State variables
    • Input variables
    • Output variables
Question: "Is this DSL Model Type instantiated by an ElmDsl object?"
Answer: YES ‚Üí Main Model

Sub-block Identification

üîß How to Identify a Sub-block

Sub-block Usage

DSL Model Type (BlkDef) ‚Üí Block Reference (BlkRef) ‚Üí Parent Block Diagram

Key Identification Criteria:

  • Embedded Usage: Referenced within another DSL Model Type's block diagram
  • Block Reference: Included via Block Reference (BlkRef) object
  • Partial Functionality: Represents only part of Main Model's functionality
  • No Self-Initialization: Typically lacks relevant initial conditions
  • Hierarchical Position: Not the top-level model
Question: "Is this DSL Model Type referenced by a BlkRef within another model?"
Answer: YES ‚Üí Sub-block

Purpose: Modular building blocks for complex hierarchical models

Primary Identification:

DSL Model Type (BlkDef) ‚Üí Basic Options Tab ‚Üí "Macro" Flag = ‚úì CHECKED

DSL Macro Characteristics:

  • Specific Sub-block Type: Special category of sub-block
  • Lowest-Level Building Blocks: Basic reusable functions
  • Common Functions: Gains, filters, integrators, limiters
  • Maximum Reusability: Designed for multiple uses
  • Typically Code-Based: Pure DSL code preferred over graphical

Usage Context:

  • Block Reference Integration: Used via BlkRef in higher-level diagrams
  • Multi-Output Handling: All defined outputs must be accounted for
  • Library Distribution: Often stored in libraries for sharing

Why Code-Based? Graphical macros reduce reusability and flexibility

Practical Implementation

Setting Up Model Hierarchy:

1. Create DSL Macros (Building Blocks)

  • Create DSL Model Type (BlkDef)
  • Check "Macro" flag in Basic Options
  • Define using DSL code (preferred) or graphically
  • Do NOT include initialization conditions

2. Create Sub-blocks (if needed)

  • Create DSL Model Type (BlkDef)
  • Leave "Macro" flag unchecked
  • Use Block References to include macros
  • Handle partial model functionality

3. Create Main Model

  • Create DSL Model Type (BlkDef) for top level
  • Include sub-blocks/macros via Block References
  • Define complete initialization conditions
  • Instantiate as DSL Model (ElmDsl) in network

Key Insight: The ElmDsl object is just an instance of the Main Model for simulation

Input/Output Configuration

Configuration Requirements Based on Identified Role

Main Model Requirements:

üì• Inputs

  • Initialization REQUIRED
  • Can be done within model or externally
  • Unconnected inputs need start values
  • Missing start values trigger errors

üì§ Outputs

  • Initialization REQUIRED
  • Internal or external initialization
  • Must define steady-state values
  • Based on load flow calculation

Sub-block & DSL Macro Requirements:

Inputs & Outputs: Typically do NOT contain initial conditions
Reason: Provides flexibility for higher-level initialization
Exception: All defined outputs must be accounted for in Block Reference calls

Macro Definition: 3 outputs defined
Block Reference Call: Must list all 3 outputs (even if unused)

Initialization Methods

1. Direct Initialization (Equations Page)

inc(varnm) = expr

Evaluated when the model is reset - defines initial conditions directly

2. Automatic Initialization

  • If inc(varnm) not defined, normal assignment evaluated
  • Uses start values for unconnected inputs
  • Based on steady-state operating point

3. DPL-based Configuration Script

  • Optional script for Top-level models
  • Executed after load flow, before initialization
  • Overwrites initial parameter values
  • Provides flexible parameter configuration

Purpose: Determine steady-state operating point before dynamic simulation starts

Graphical I/O Definition

Boundary Frame Structure

Left Side: Input Signals ‚Üí [DSL Model Logic] ‚Üí Right Side: Output Signals

Automatic Signal Detection:

  • Input Variables: Signal lines starting from left side of Boundary frame
  • Output Variables: Signal lines ending on right side of Boundary frame
  • Automatic Recognition: PowerFactory identifies I/O from diagram structure

Important Notes:

  • All defined outputs must be accounted for in macro calls
  • Macro definition specifies available outputs
  • Unused outputs still need to be listed in calls
  • Initialization logic handled at Main Model level

DSL Development Process

Enhanced 6-Step Process:

Step 1: Define Model Hierarchy

Determine if creating Main Model or DSL Macro

Step 2: Create DSL Model Type (BlkDef)

Set appropriate flags (Macro flag for sub-blocks)

Step 3: Choose Definition Method

Select between graphical block diagram or direct code approach

Step 4: Build the Block Diagram

  • Open block diagram editor
  • Define Boundary frame for I/O
  • Insert DSL Macros as Block References
  • Connect blocks and define signal flow

Step 5: Configure Inputs/Outputs

Apply appropriate initialization based on model type

Step 6: Implement Logic & Initialize

Use built-in DSL functions and define initial conditions

Best Practices

Model Design:

  • Plan Hierarchy: Decide Main Model vs DSL Macro early
  • Initialize Properly: Follow initialization rules for model type
  • Handle Unconnected Inputs: Always provide start values
  • Document I/O: Clearly define input/output purposes

Development Workflow:

  • Start with Macros: Build reusable components first
  • Test Incrementally: Validate each level of hierarchy
  • Use Configuration Scripts: For flexible parameter management
  • Leverage Load Flow: Base initialization on steady-state

Common Pitfall: Forgetting to initialize inputs/outputs in Main Models leads to simulation errors
